# Provenance Guard — Project Planning

## Overview

Provenance Guard is a Flask-based API that estimates whether a piece of text was written by a human or generated by an AI model. It combines two independent signals — an LLM-based analysis via Groq and a local stylometric analysis — into a single weighted confidence score, surfaces a transparency label for non-technical readers, and provides an appeals workflow for disputed classifications.

---

## Detection Signals

Two signals:

1. **LLM-based analysis via Groq (llama-3.3-70b-versatile):** sends submitted text to the model with instructions to evaluate writing style, consistency, phrasing, tone, and likelihood of AI generation. Returns an AI-likelihood score between 0 and 1, a confidence value, and a short explanation. Strength: understands context and can recognize sophisticated AI writing. Weakness: can be confidently wrong, not fully deterministic, output can shift between runs.

2. **Stylometric analysis:** pure Python, no external model. Computes sentence length variance, type-token ratio (vocabulary diversity), and punctuation density, combined into a numerical AI-likelihood score between 0 and 1. Strength: deterministic, fast, explainable, independent of any LLM. Weakness: short texts provide little signal, and skilled human writers can resemble AI by these metrics alone.

---

## Combining Signals

Final confidence = 0.6 × Groq score + 0.4 × stylometric score. Groq gets more weight because it understands meaning and context; stylometrics gets less weight because it is a purely structural signal blind to content. Weights are adjustable after testing against real inputs.

---

## Uncertainty Representation

Final score is a probability from 0.0 (very likely human) to 1.0 (very likely AI). Three label bands:

- **0.00–0.30:** Likely Human-Written
- **0.31–0.69:** Uncertain — Unable to Determine Reliably
- **0.70–1.00:** Likely AI-Generated

Every response includes the line: "This result is an estimate, not proof. False positives and false negatives are possible."

---

## Transparency Label Variants

Exact strings shown to a non-technical reader:

- **High-confidence AI:** "Likely AI-Generated — our system estimates this content was most likely produced by an AI model. This result is an estimate, not proof."
- **High-confidence human:** "Likely Human-Written — our system estimates this content was most likely written by a human. This result is an estimate, not proof."
- **Uncertain:** "Uncertain — Unable to Determine Reliably. Our system could not confidently determine whether this was AI or human written. This result is an estimate, not proof."

---

## Appeals Workflow

A creator who disagrees with their classification can submit an appeal containing: the original text, the creator's explanation/reasoning, and optionally supporting evidence (draft history, timestamps, notes). The appeal is logged alongside the original decision, and the content's status is updated to "under_review." A human reviewer, not an automated process, makes the final call. If overturned, the result is updated and the decision is recorded as overturned in the audit log for future calibration. No automated re-classification is implemented.

---

## Anticipated Edge Cases

1. A university student who carefully revises their essay over hours produces grammatically consistent, well-organized writing. Both Groq and stylometrics may read this polish as AI-generated, producing a false positive. This is the primary scenario the appeal process exists to catch.

2. Very short text submissions (a sentence or two) give stylometric analysis almost nothing to measure (no meaningful sentence-length variance with only one or two sentences), making that signal close to noise and overweighting Groq's read by default.

## Architecture

### Submission Flow

```
POST /submit
     |
     | raw text
     v
Groq Signal (llama-3.3-70b-versatile)
     |
     | groq_score (0–1), explanation
     v
Stylometric Signal (local Python)
     |
     | stylometric_score (0–1)
     v
Weighted Confidence Scoring
(0.6 * groq_score + 0.4 * stylometric_score)
     |
     | combined_score (0–1)
     v
Label Mapping
(Likely Human / Uncertain / Likely AI-Generated)
     |
     | label text, scores, explanation
     v
Audit Log
     |
     | logged entry (text hash, scores, label, timestamp)
     v
JSON Response
(label, combined_score, groq_score, stylometric_score, explanation, disclaimer)
```

### Appeal Flow

```
POST /appeal
     |
     | original text, creator explanation, optional evidence
     v
Status Update
(content status -> "under_review")
     |
     | submission_id, updated status
     v
Appeal Logged
(appeal stored alongside original decision)
     |
     | appeal record (original decision, creator explanation, evidence)
     v
Audit Log
     |
     | logged entry (appeal received, timestamp, reviewer pending)
     v
Confirmation Response
(status: under_review, appeal_id, next steps)
```
