# Provenance Guard

A Flask API that estimates whether a piece of text was written by a human or generated by an AI model. It combines two independent detection signals into a weighted confidence score, maps the score to a plain-language label, logs every decision to an append-only audit trail, and provides an appeal path when a creator disputes their classification.

---

## Architecture Overview

Every submission follows a single linear path through the system:

```
POST /submit
     |
     | raw text
     v
Groq Signal (llama-3.3-70b-versatile)
     |
     | groq_score (0–1), explanation
     v
Stylometric Signal (local Python)
     |
     | stylometric_score (0–1)
     v
Weighted Confidence Scoring
(0.6 * groq_score + 0.4 * stylometric_score)
     |
     | combined_score (0–1)
     v
Label Mapping
(Likely Human / Uncertain / Likely AI-Generated)
     |
     | label text, scores, explanation
     v
Audit Log  +  Submission Store
     |
     | logged entry, persisted record
     v
JSON Response
```

When a `POST /submit` arrives, the server runs both signals concurrently — the Groq API call and the local stylometric computation — then combines them using a fixed weighted formula. The result is mapped to one of three label bands. The full decision (both raw scores, the combined confidence, and the label) is appended to `audit_log.json` and the submission record is saved to `submissions.json` with a status of `classified`. The JSON response returned to the caller contains every score, the label text, and a mandatory disclaimer.

The appeal flow is separate:

```
POST /appeal
     |
     | content_id, creator_reasoning, optional evidence
     v
Submission lookup (submissions.json)
     |
     | found record with original decision
     v
Status updated to "under_review"
     |
     v
Appeal entry appended to audit_log.json
(original attribution, original confidence, reasoning, evidence)
     |
     v
Confirmation response
(content_id, status: under_review)
```

No automated re-classification happens at appeal time. The status change and the audit entry are all the system does — a human reviewer reads the log and makes the final call.

## Detection Signals

### Signal 1: Groq LLM Analysis

**What it measures:** The text is sent to `llama-3.3-70b-versatile` with a system prompt instructing the model to act as a forensic linguist. It evaluates writing style, internal consistency, phrasing patterns, and tone, then returns a structured JSON object: `ai_likelihood` (0–1), `confidence` (0–1), and a short explanation of its reasoning. Temperature is set to 0.0 to minimize run-to-run variance.

**Why it was chosen:** A large language model understands what text *means*, not just how it is structured. It can recognize the kind of smooth, hedged, transitional language that LLMs tend to produce — phrases like "it is important to note," "furthermore," and "it is equally essential to consider" — and can pick up on an absence of the false starts, contradictions, and specificity that characterize human writing. No purely statistical method can do this.

**Its blind spot:** The model can be confidently wrong. It has no ground truth — it is making a probabilistic judgment about a population of texts, and any individual piece of writing can fall outside what it expects. Two identical runs on the same text can produce slightly different scores. The model is also vulnerable to adversarial inputs: a human writer who deliberately mimics AI phrasing may fool it, and an AI output that was heavily edited may not trigger it.

---

### Signal 2: Stylometric Analysis

**What it measures:** Three structural properties of the text, computed entirely in Python with no external dependencies:

- **Sentence length variance:** The text is split on `.`, `!`, and `?`. Word counts per sentence are computed, and the population variance of those counts is the signal. AI-generated text tends to produce sentences of similar length; human writing, especially informal writing, varies more.
- **Type-token ratio (TTR):** Unique words divided by total words, after lowercasing and stripping punctuation. A low TTR means the writer reuses the same vocabulary frequently — a pattern more common in AI outputs tuned for consistency. A high TTR suggests richer, more varied vocabulary.
- **Punctuation density:** Punctuation characters divided by total characters. AI text tends to cluster near a moderate, consistent density. Very low density (informal text with few commas or periods) or very high density (dense academic prose) is more characteristic of human writers.

Each sub-metric is normalized to [0, 1] and combined into a single `stylometric_score` (0 = human-like, 1 = AI-like), with sentence variance weighted most heavily (0.50), TTR next (0.35), and punctuation density least (0.15).

**Why it was chosen:** It is deterministic, fast, and requires no API call. It provides an independent check on the Groq signal — if both agree, confidence in the combined score is higher. If they diverge, the uncertain band absorbs the ambiguity rather than forcing a false commitment to either label.

**Its blind spot:** Structure is not meaning. A careful human writer who produces grammatically clean, consistently-paced sentences can look identical to an AI by these metrics. Short texts (under two or three sentences) provide almost no useful signal — sentence variance is undefined with a single sentence, and TTR is unreliable on fewer than ~50 tokens. In those cases, `stylometric_signal` returns a neutral score of 0.5 and notes the limitation.

## Confidence Scoring

The final confidence score is computed as:

```
confidence = 0.6 × groq_score + 0.4 × stylometric_score
```

Groq receives the higher weight (0.6) because it reasons about content and meaning — it understands that "Furthermore, stakeholders across various sectors must collaborate" reads differently from "probably wont go back unless someone drags me there." Stylometrics receives the lower weight (0.4) because it is a structural backstop: useful as corroboration, but blind to what the text is actually saying.

The weights were set by design before any test data was collected and are explicitly flagged as adjustable in the project spec after further calibration.

### Real examples

**Higher confidence — clearly AI-generated:**

> "Artificial intelligence represents a transformative paradigm shift in modern society. It is important to note that while the benefits of AI are numerous, it is equally essential to consider the ethical implications. Furthermore, stakeholders across various sectors must collaborate to ensure responsible deployment."

| Signal | Score |
|---|---|
| `groq_score` | 0.80 |
| `stylometric_score` | 0.3327 |
| `confidence` | **0.6131** |
| Label | Uncertain — Unable to Determine Reliably |

Groq correctly reads the hedged corporate register and transitional phrasing as AI-typical. Stylometrics is more conservative (0.33) because the three sentences have meaningful variance in length and the vocabulary is reasonably diverse. The weighted combination lands at 0.61 — still in the Uncertain band rather than crossing the 0.70 threshold into Likely AI-Generated. This is a real limitation discussed further in Known Limitations.

**Borderline — formally-written human text:**

> "The relationship between monetary policy and asset price inflation has been extensively studied in the literature. Central banks face a fundamental tension between their mandate for price stability and the unintended consequences of prolonged low interest rates on equity and real estate valuations."

| Signal | Score |
|---|---|
| `groq_score` | 0.70 |
| `stylometric_score` | 0.3233 |
| `confidence` | **0.5493** |
| Label | Uncertain — Unable to Determine Reliably |

Groq is suspicious of the formal academic register (0.70), but stylometrics reads the text as relatively human-like (0.32) — the two sentences differ meaningfully in length and the vocabulary is non-repetitive. The combined score of 0.55 sits near the middle of the Uncertain band. This is the false positive scenario planning.md anticipated: polished human writing can look like AI to a language model. The Uncertain label is the correct response here — the system correctly avoids claiming certainty it does not have.

## Transparency Label

Every `/submit` response includes a `label` field with one of the following strings, determined by the combined confidence score:

| Score range | `attribution` | Label text |
|---|---|---|
| 0.00 – 0.30 | `likely_human` | "Likely Human-Written — our system estimates this content was most likely written by a human. This result is an estimate, not proof." |
| 0.31 – 0.69 | `uncertain` | "Uncertain — Unable to Determine Reliably. Our system could not confidently determine whether this was AI or human written. This result is an estimate, not proof." |
| 0.70 – 1.00 | `likely_ai` | "Likely AI-Generated — our system estimates this content was most likely produced by an AI model. This result is an estimate, not proof." |

Every response also includes a standalone `disclaimer` field:

> "This result is an estimate, not proof. False positives and false negatives are possible."

This disclaimer is not optional — it is present on every response regardless of confidence, including high-confidence classifications.

## Rate Limiting

`POST /submit` is limited to **10 requests per minute and 100 requests per day** per IP address, enforced by Flask-Limiter using in-memory storage.

The per-minute limit is designed to allow realistic single-creator usage — a person submitting several drafts of a document in a session — while blocking scripted bulk submissions that could be used to probe the system for adversarial inputs or exhaust Groq API quota. 10/minute is generous for interactive use and tight for automation. The daily cap of 100 provides a secondary ceiling against sustained abuse across a longer window.

`GET /log` and `POST /appeal` are not rate-limited. Appeals are expected to be infrequent and human-initiated; the log endpoint is read-only and carries no external API cost.

## Known Limitations

**The clearly-AI calibration text only reached 0.61 combined confidence.** The submission "Artificial intelligence represents a transformative paradigm shift in modern society..." was chosen as an "easy" AI-detection case — hedged phrasing, transitional connectives, no specificity. Groq correctly scored it at 0.80, but the stylometric signal pulled it down to 0.33 (short text, reasonable sentence-length variance, decent vocabulary diversity), landing the combined score at 0.61. That is in the Uncertain band, not Likely AI-Generated. For an easy case, 0.61 is softer than expected. This suggests either the 0.6/0.4 weighting undervalues the Groq signal on short texts where stylometrics is noisy, or that the Groq prompt could be tuned to respond more strongly to this class of corporate-register AI writing. More calibration data with known-AI and known-human ground truth is needed before adjusting the weights.

**Stylometric analysis is unreliable on short texts.** Sentence-length variance requires at least two or three sentences to be meaningful. With a single sentence there is no variance to compute — the function defaults to a neutral score of 0.5 and notes the edge case. In practice this means any one- or two-sentence submission effectively receives only the Groq signal (since the stylometric contribution is a coin flip). The system does not warn the caller when this happens, which is a transparency gap worth closing.

**In-memory rate limiting does not survive restarts.** Flask-Limiter is configured with `storage_uri="memory://"`, which means the per-IP counters are lost when the process restarts. In a production deployment with process restarts or multiple workers, this limit would not be enforced correctly. A Redis-backed storage URI would fix this.

## Spec Reflection

**Where planning.md helped:** Writing the exact label text and confidence thresholds in the spec before any code was written meant there was no ambiguity when implementing `map_attribution()`. The three label strings and the 0.00/0.31/0.70 band boundaries were already decided — the function was essentially transcription. If those decisions had been deferred to implementation time, they likely would have been revised mid-milestone as edge cases surfaced, causing inconsistencies between the API response and any documentation written afterward.

**Where the spec held up without divergence:** The planned architecture (two independent signals, weighted combination, flat JSON audit log, human-review-only appeal resolution) was implemented as written without modification. The one area where reality added nuance not in the spec is the short-text edge case in stylometric analysis — the spec notes the limitation but doesn't specify what to return. The implementation defaults to 0.5 and flags it in the return dict, which is a reasonable conservative choice but not one the spec mandated.

## AI Usage

**Signal normalization in `stylometric_signal`:** The stylometric function was directed to compute sentence variance, TTR, and punctuation density and combine them into a [0, 1] score. The initial output used a simple linear normalization for sentence variance capped at a fixed ceiling. After reviewing test results, the punctuation density sub-metric was revised: the original implementation treated low punctuation as neutral, but the calibration runs showed informal human text (very low punctuation density) was being penalized. The normalization was replaced with a tent function centered at 0.07 — the value where AI-typical density peaks — so that both very low and very high punctuation density push the score toward human-like rather than both pushing toward AI-like.

**System prompt design for the Groq signal:** The initial system prompt asked the model to "determine if this text is AI-generated." This produced outputs that were often binary or reluctant to return well-formed JSON when the text was ambiguous. The prompt was revised to frame the model as a "forensic linguist" evaluating specific named dimensions (style, consistency, phrasing, tone), and the output format was made explicit with a typed example in the prompt itself. Temperature was set to 0.0 after observing that the same text produced slightly different scores across runs at the default temperature. The combination of role framing, named evaluation criteria, and strict format instruction produced consistently parseable JSON across all calibration inputs.
